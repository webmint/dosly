# Data Model: Settings Domain Purify

This spec changes the shape of one entity (`AppSettings`) and introduces one new domain enum (`AppThemeMode`). No persistent storage layout changes beyond the `themeMode` key's value-type (int → string). All changes are local-only — there are no remote APIs.

## Entities

### `AppThemeMode` (NEW)

`lib/features/settings/domain/entities/app_theme_mode.dart`. Pure-Dart enum.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `code` | `String` | yes | Stable IETF-style code persisted to SharedPreferences. Order-independent across SDK changes |

| Value | `code` | Description |
|-------|--------|-------------|
| `AppThemeMode.light` | `'light'` | Default; light Material 3 theme |
| `AppThemeMode.dark` | `'dark'` | Dark Material 3 theme |

**Static helper**: `AppThemeMode fromCodeOrDefault(String? code)` — returns `AppThemeMode.light` for any unknown / null / empty code. Same defensive pattern as `AppLanguage.values.firstWhere(... orElse: AppLanguage.en)` already used at `settings_local_data_source.dart:72–75`.

**Validation rules**:
- `code` is one of `'light'` or `'dark'` — enforced by the enum itself; impossible to construct an invalid value
- Anything else read from disk maps to `AppThemeMode.light`

**Mapping at the presentation seam** (`lib/app.dart` only):
- `AppThemeMode.light` → `Flutter.ThemeMode.light`
- `AppThemeMode.dark` → `Flutter.ThemeMode.dark`
- `useSystemTheme: true` short-circuits to `Flutter.ThemeMode.system` before mapping

The mapping is exhaustive over `AppThemeMode`'s 2 values; the Dart compiler enforces it because `AppThemeMode` has no `system` value to forget.

### `AppSettings` (CHANGED)

`lib/features/settings/domain/entities/app_settings.dart`. Becomes a `@freezed class`.

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `useSystemTheme` | `bool` | no | `true` | Follow device theme when `true`; consult `manualThemeMode` when `false` |
| `manualThemeMode` | `AppThemeMode` | no | `AppThemeMode.light` | Manual override consulted when `useSystemTheme == false` |
| `useSystemLanguage` | `bool` | no | `true` | Follow device language when `true`; consult `manualLanguage` when `false` |
| `manualLanguage` | `AppLanguage` | no | `AppLanguage.en` | Manual override consulted when `useSystemLanguage == false` |

Generated by `freezed`:
- `==` / `hashCode` (value-equality across all four fields)
- `toString` (deterministic, useful for debugging — but constitution §4.2.1 forbids `print`/`debugPrint` in committed code, so debug printing of the value is not a sanctioned use)
- `copyWith({useSystemTheme, manualThemeMode, useSystemLanguage, manualLanguage})` — all parameters optional and nullable for the standard "preserve original on null" semantic

**Removed** in this spec:
- `effectiveThemeMode` getter (`ThemeMode get effectiveThemeMode => useSystemTheme ? ThemeMode.system : manualThemeMode;`) — moves into `lib/app.dart` as inline computation.
- `effectiveLocale` getter (`Locale? get effectiveLocale => useSystemLanguage ? null : Locale(manualLanguage.code);`) — moves into `lib/app.dart` as inline computation.

### `AppLanguage` (UNCHANGED)

Existing `lib/features/settings/domain/entities/app_language.dart` is unchanged by this spec. Recorded here for completeness:

| Value | `code` | `nativeName` |
|-------|--------|--------------|
| `AppLanguage.en` | `'en'` | `'English'` |
| `AppLanguage.de` | `'de'` | `'Deutsch'` |
| `AppLanguage.uk` | `'uk'` | `'Українська'` |

## Relationships

- `AppSettings.manualThemeMode → AppThemeMode` (composition; one-to-one)
- `AppSettings.manualLanguage → AppLanguage` (composition; one-to-one — already exists)

## Persistence Layout

| Key | Was | Becomes | Notes |
|-----|-----|---------|-------|
| `themeMode` | `int` (`ThemeMode.index`, e.g. `1` for `light`) | `String` (`'light'` / `'dark'`) | Order-independent; matches `docs/features/settings.md:158` documentation; legacy int reads return `null` from `getString` and fall back to `AppThemeMode.light` |
| `useSystemTheme` | `bool` | `bool` | Unchanged |
| `useSystemLanguage` | `bool` | `bool` | Unchanged |
| `manualLanguage` | `String` (e.g. `'en'`) | `String` (unchanged) | Unchanged |

`SharedPreferencesWithCache.allowList` in `lib/main.dart:12–17` is unchanged — same four key names.

## Out of Model

- No remote API (`/specify`'s overall scope rules out backend per constitution §1).
- No JSON serialization for `AppSettings` (each field is persisted under its own SharedPreferences key by `SettingsLocalDataSource`, not as a JSON blob). Therefore `freezed`'s `fromJson`/`toJson` codegen is **not** activated for this entity — `json_annotation`/`json_serializable` packages are NOT added.
- No new `Failure` variants. `CacheFailure(String message)` remains the only concrete subclass. Bug 006 (Failure hierarchy completion) is a separate spec.
