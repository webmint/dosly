# Internationalization (i18n)

## Overview

dosly ships with three locales ‚Äî **English** (`en`, the default fallback), **German** (`de`), and **Ukrainian** (`uk`). Translations are sourced from ARB files under `lib/l10n/`, compiled into a generated `AppLocalizations` class by `flutter gen-l10n`, and consumed in widgets via a single `context.l10n` extension. The active locale is resolved from the device locale on every launch; there is no in-app language picker (that ships with the future Settings feature).

Currently four strings are translated: the Settings icon tooltip and the three bottom-navigation destination labels (Today, Meds, History).

Historical context: [spec](../../specs/006-i18n-support/spec.md) ¬∑ [plan](../../specs/006-i18n-support/plan.md) ¬∑ [review](../../specs/006-i18n-support/review.md)

## Architecture

```
lib/l10n/
‚îú‚îÄ‚îÄ app_en.arb                  # Template ARB ‚Äî English (source of truth for keys + @metadata)
‚îú‚îÄ‚îÄ app_de.arb                  # German translations
‚îú‚îÄ‚îÄ app_uk.arb                  # Ukrainian translations

‚îú‚îÄ‚îÄ app_localizations_en.dart   # Generated: English subclass
‚îú‚îÄ‚îÄ app_localizations_de.dart   # Generated: German subclass
‚îú‚îÄ‚îÄ app_localizations_uk.dart   # Generated: Ukrainian subclass

```

**Data flow**: ARB files ‚Üí `flutter gen-l10n` (runs on `flutter pub get`) ‚Üí `AppLocalizations` class ‚Üí `context.l10n` ‚Üí widget `build()` methods.

`AppLocalizations` is wired into `MaterialApp.router` in `lib/app.dart`:

```dart
MaterialApp.router(
  localizationsDelegates: AppLocalizations.localizationsDelegates,
  supportedLocales: AppLocalizations.supportedLocales,
  localeResolutionCallback: _resolveLocale,
  ...
)
```

### The `context.l10n` extension

`lib/l10n/l10n_extensions.dart` defines one extension getter:

```dart
extension AppLocalizationsContext on BuildContext {
  AppLocalizations get l10n => AppLocalizations.of(this)!;
}
```

This is the **only place in the codebase where `AppLocalizations.of(context)!` is called**. The `!` is a constitution-documented exception (spec ¬ß7): the value is non-null whenever delegates are correctly registered, and centralizing the assertion here means zero `!` at every widget call site. Any future "framework returns nullable but never is in practice" primitive should follow the same pattern.

Widget usage:

```dart
// lib/features/home/presentation/widgets/home_bottom_nav.dart
final l = context.l10n;
NavigationDestination(icon: const Icon(LucideIcons.house), label: l.bottomNavToday),
```

### Locale fallback

Flutter's default `BasicLocaleListResolutionCallback` falls back to the alphabetically-first entry of `supportedLocales`. Because `gen_l10n` emits the list as `[de, en, uk]`, the default would incorrectly surface German to users on unsupported locales (e.g., French, Japanese). The private `_resolveLocale` function in `lib/app.dart` overrides this: it matches by `languageCode` and returns `Locale('en')` when no match is found.

```dart
// lib/app.dart
Locale _resolveLocale(Locale? deviceLocale, Iterable<Locale> supportedLocales) {
  if (deviceLocale != null) {
    for (final supported in supportedLocales) {
      if (supported.languageCode == deviceLocale.languageCode) return supported;
    }
  }
  return const Locale('en');
}
```

### Generated files

`l10n.yaml` is configured with no `synthetic-package` key, which defaults to the modern Flutter behavior (`synthetic-package: false`). Generated files live at `lib/l10n/app_localizations*.dart` and are **committed to Git** so that fresh clones compile without running `flutter pub get` first. This is consistent with how other codegen outputs (freezed, drift, riverpod) will be handled in this project. Running `flutter pub get` or `flutter gen-l10n` regenerates them; the `dart analyze` PostToolUse hook will flag stale generated code if an ARB key is added without regenerating.

Widgets import the generated class via the normal package-relative path:

```dart
import 'package:dosly/l10n/app_localizations.dart';
```

(Not `package:flutter_gen/gen_l10n/...` ‚Äî that path belongs to the deprecated `synthetic-package: true` setup.)

## How to add a new translated string

1. **Edit `lib/l10n/app_en.arb`** ‚Äî add the key, its English value, and a `@key` metadata block:

```json
{
  "myNewString": "Do the thing",
  "@myNewString": {
    "description": "Button label on FooScreen that triggers the thing."
  }
}
```

2. **Edit `lib/l10n/app_de.arb` and `lib/l10n/app_uk.arb`** ‚Äî add the same key with translated values (no `@` metadata needed):

```json
{
  "myNewString": "Die Sache tun"
}
```

3. **Regenerate** by running `flutter pub get` (codegen runs implicitly) or `flutter gen-l10n` explicitly. The generated files in `lib/l10n/app_localizations*.dart` update automatically.

4. **Consume in a widget** via `context.l10n`:

```dart
Text(context.l10n.myNewString)
```

5. **Commit** the ARB files and the regenerated `lib/l10n/app_localizations*.dart` files together.

> Every ARB file must contain every key. If you add a key to `app_en.arb` but omit it from `app_de.arb`, `flutter gen-l10n` will error. The English value is used as the fallback only when the key is genuinely absent from the template, which is not a valid state.

## How to add a new locale

1. Create `lib/l10n/app_xx.arb` (replace `xx` with the BCP 47 language tag) containing `"@@locale": "xx"` and every existing key translated:

```json
{
  "@@locale": "fr",
  "settingsTooltip": "Param√®tres",
  "bottomNavToday": "Aujourd'hui",
  "bottomNavMeds": "M√©dicaments",
  "bottomNavHistory": "Historique"
}
```

2. Run `flutter pub get` or `flutter gen-l10n` ‚Äî a new `app_localizations_xx.dart` subclass is generated and `AppLocalizations.supportedLocales` automatically includes the new locale.

3. No changes to `lib/app.dart` are required ‚Äî `_resolveLocale` and `AppLocalizations.supportedLocales` pick up the new locale automatically.

4. Commit all four files: the new ARB, the updated `app_localizations.dart`, and the new `app_localizations_xx.dart`.

## What is NOT translated (and why)

| String | Location | Reason |
|---|---|---|
| `'Dosly'` / `'dosly'` | AppBar title, `MaterialApp.title` | Brand name ‚Äî intentionally language-neutral |
| `'Hello World'` | `HomeScreen` body placeholder | Temporary; will be replaced wholesale when the Today feature ships |
| `'Theme preview'` / `'dosly ¬∑ M3 preview'` | `ThemePreviewScreen`, `HomeScreen` button | Dev scaffolding scheduled for removal post-MVP |
| `'Demo FAB'` / `'Cycle theme mode'` | `ThemePreviewScreen` | Same ‚Äî dev-only |

Translating any of these would produce strings that get deleted when the real content lands, which is wasted work.

## Rules

- **Always use `context.l10n`** to read localized strings. Never call `AppLocalizations.of(context)` or `AppLocalizations.of(context)!` directly in a widget ‚Äî the extension is the only sanctioned path.
- **Never import `AppLocalizations` in `domain/`**. Localization is a presentation concern. Constitution ¬ß2.1 forbids `package:flutter/*` in the domain layer; `AppLocalizations` follows the same boundary.
- **Keep ARBs in sync**: if you edit one locale's ARB, update all others in the same commit/PR. Stale translations compile but ship incorrect UI.
- **No locale-aware state yet**: there is no `localeController` analogue to `themeController`. The device locale is the sole driver. A reactive locale picker will arrive with the Settings feature.

## Out of scope (deferred)

These topics are explicitly not covered by this feature and will require separate specs:

- In-app language picker UI and persistence
- Locale-aware date/number formatting (`DateFormat`, `NumberFormat` from `intl`)
- Pluralization and ICU message syntax in ARB files
- Right-to-left (RTL) layout support
- Additional locales beyond `en`, `de`, `uk`
- Notification text translation
