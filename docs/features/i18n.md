# Internationalization (i18n)

## Overview

dosly ships with three locales — **English** (`en`, the default fallback), **German** (`de`), and **Ukrainian** (`uk`). Translations are sourced from ARB files under `lib/l10n/`, compiled into a generated `AppLocalizations` class by `flutter gen-l10n`, and consumed in widgets via a single `context.l10n` extension.

The active locale is driven by user preference. When the user opts in to the system language (the default), the device locale is resolved via `localeResolutionCallback`. When the user pins the app to a specific language via the Settings screen, `MaterialApp.locale` is set to that locale directly and the callback is bypassed.

Currently translated groups: the Settings screen controls (title, tooltip, section headers, toggle labels), the three bottom-navigation destination labels, and theme-selector labels.

Historical context: [spec](../../specs/006-i18n-support/spec.md) · [plan](../../specs/006-i18n-support/plan.md) · [review](../../specs/006-i18n-support/review.md)

## Architecture

```
lib/l10n/
├── app_en.arb                  # Template ARB — English (source of truth for keys + @metadata)
├── app_de.arb                  # German translations
├── app_uk.arb                  # Ukrainian translations
├── app_localizations.dart      # Generated: AppLocalizations base class + delegates
├── app_localizations_en.dart   # Generated: English subclass
├── app_localizations_de.dart   # Generated: German subclass
├── app_localizations_uk.dart   # Generated: Ukrainian subclass
└── l10n_extensions.dart        # BuildContext.l10n extension — the only sanctioned consumer API

l10n.yaml                       # gen-l10n config (repo root)
```

**Data flow**: ARB files → `flutter gen-l10n` (runs on `flutter pub get`) → `AppLocalizations` class → `context.l10n` → widget `build()` methods.

`AppLocalizations` is wired into `MaterialApp.router` in `lib/app.dart`:

```dart
MaterialApp.router(
  localizationsDelegates: AppLocalizations.localizationsDelegates,
  supportedLocales: AppLocalizations.supportedLocales,
  localeResolutionCallback: _resolveLocale,
  locale: ref.watch(settingsProvider.select((s) => s.effectiveLocale)),
  ...
)
```

`locale` is driven by `AppSettings.effectiveLocale`:

- When the user is in **system language mode** (the default), `effectiveLocale` is `null`. Flutter invokes `localeResolutionCallback` (`_resolveLocale`) to match the device locale against `AppLocalizations.supportedLocales`, falling back to `Locale('en')` when no match is found.
- When the user has **manually selected** a language, `effectiveLocale` is a non-null `Locale` (e.g. `Locale('de')`). Flutter uses it directly and does **not** invoke `localeResolutionCallback`.

This means adding a new supported locale requires updating both the ARB files **and** `AppLanguage` — the enum is the authoritative list of languages the language picker exposes. `_resolveLocale` and `AppLocalizations.supportedLocales` continue to work without changes.

### The `context.l10n` extension

`lib/l10n/l10n_extensions.dart` defines one extension getter:

```dart
extension AppLocalizationsContext on BuildContext {
  AppLocalizations get l10n => AppLocalizations.of(this)!;
}
```

This is the **only place in the codebase where `AppLocalizations.of(context)!` is called**. The `!` is a constitution-documented exception (spec §7): the value is non-null whenever delegates are correctly registered, and centralizing the assertion here means zero `!` at every widget call site. Any future "framework returns nullable but never is in practice" primitive should follow the same pattern.

Widget usage:

```dart
// lib/core/widgets/app_bottom_nav.dart
final l = context.l10n;
NavigationDestination(icon: const Icon(LucideIcons.house), label: l.bottomNavToday),
```

### Locale fallback

Flutter's default `BasicLocaleListResolutionCallback` falls back to the alphabetically-first entry of `supportedLocales`. Because `gen_l10n` emits the list as `[de, en, uk]`, the default would incorrectly surface German to users on unsupported locales (e.g., French, Japanese). The private `_resolveLocale` function in `lib/app.dart` overrides this: it matches by `languageCode` and returns `Locale('en')` when no match is found.

```dart
// lib/app.dart
Locale _resolveLocale(Locale? deviceLocale, Iterable<Locale> supportedLocales) {
  if (deviceLocale != null) {
    for (final supported in supportedLocales) {
      if (supported.languageCode == deviceLocale.languageCode) return supported;
    }
  }
  return const Locale('en');
}
```

### Generated files

`l10n.yaml` is configured with no `synthetic-package` key, which defaults to the modern Flutter behavior (`synthetic-package: false`). Generated files live at `lib/l10n/app_localizations*.dart` and are **committed to Git** so that fresh clones compile without running `flutter pub get` first. This is consistent with how other codegen outputs (freezed, drift, riverpod) will be handled in this project. Running `flutter pub get` or `flutter gen-l10n` regenerates them; the `dart analyze` PostToolUse hook will flag stale generated code if an ARB key is added without regenerating.

Widgets import the generated class via the normal package-relative path:

```dart
import 'package:dosly/l10n/app_localizations.dart';
```

(Not `package:flutter_gen/gen_l10n/...` — that path belongs to the deprecated `synthetic-package: true` setup.)

## How to add a new translated string

1. **Edit `lib/l10n/app_en.arb`** — add the key, its English value, and a `@key` metadata block:

```json
{
  "myNewString": "Do the thing",
  "@myNewString": {
    "description": "Button label on FooScreen that triggers the thing."
  }
}
```

2. **Edit `lib/l10n/app_de.arb` and `lib/l10n/app_uk.arb`** — add the same key with translated values (no `@` metadata needed):

```json
{
  "myNewString": "Die Sache tun"
}
```

3. **Regenerate** by running `flutter pub get` (codegen runs implicitly) or `flutter gen-l10n` explicitly. The generated files in `lib/l10n/app_localizations*.dart` update automatically.

4. **Consume in a widget** via `context.l10n`:

```dart
Text(context.l10n.myNewString)
```

5. **Commit** the ARB files and the regenerated `lib/l10n/app_localizations*.dart` files together.

> Every ARB file must contain every key. If you add a key to `app_en.arb` but omit it from `app_de.arb`, `flutter gen-l10n` will error. The English value is used as the fallback only when the key is genuinely absent from the template, which is not a valid state.

## How to add a new locale

1. Create `lib/l10n/app_xx.arb` (replace `xx` with the BCP 47 language tag) containing `"@@locale": "xx"` and every existing key translated:

```json
{
  "@@locale": "fr",
  "settingsTooltip": "Paramètres",
  "bottomNavToday": "Aujourd'hui",
  "bottomNavMeds": "Médicaments",
  "bottomNavHistory": "Historique"
}
```

2. Run `flutter pub get` or `flutter gen-l10n` — a new `app_localizations_xx.dart` subclass is generated and `AppLocalizations.supportedLocales` automatically includes the new locale.

3. No changes to `lib/app.dart` are required — `_resolveLocale` and `AppLocalizations.supportedLocales` pick up the new locale automatically.

4. Commit all four files: the new ARB, the updated `app_localizations.dart`, and the new `app_localizations_xx.dart`.

## What is NOT translated (and why)

| String | Location | Reason |
|---|---|---|
| `'Dosly'` / `'dosly'` | AppBar title, `MaterialApp.title` | Brand name — intentionally language-neutral |
| `'Hello World'` | `HomeScreen` body placeholder | Temporary; will be replaced wholesale when the Today feature ships |
| `'Theme preview'` / `'dosly · M3 preview'` | `ThemePreviewScreen`, `HomeScreen` button | Dev scaffolding scheduled for removal post-MVP |
| `'Demo FAB'` / `'Cycle theme mode'` | `ThemePreviewScreen` | Same — dev-only |

Translating any of these would produce strings that get deleted when the real content lands, which is wasted work.

## Rules

- **Always use `context.l10n`** to read localized strings. Never call `AppLocalizations.of(context)` or `AppLocalizations.of(context)!` directly in a widget — the extension is the only sanctioned path.
- **Never import `AppLocalizations` in `domain/`**. Localization is a presentation concern. Constitution §2.1 forbids `package:flutter/*` in the domain layer; `AppLocalizations` follows the same boundary.
- **Keep ARBs in sync**: if you edit one locale's ARB, update all others in the same commit/PR. Stale translations compile but ship incorrect UI.
- **Locale state lives in `settingsProvider`**: `AppSettings.effectiveLocale` is the single source of truth. Never read `Localizations.localeOf(context)` to determine the user's language choice — that reflects what Flutter resolved after applying the setting, which may differ during a transition. The provider is the canonical source.

## Out of scope (deferred)

These topics are not yet covered and will require separate specs:

- Locale-aware date/number formatting (`DateFormat`, `NumberFormat` from `intl`)
- Pluralization and ICU message syntax in ARB files
- Right-to-left (RTL) layout support
- Additional locales beyond `en`, `de`, `uk`
- Notification text translation
